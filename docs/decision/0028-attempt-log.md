# ADR-0028:試行ごとの記録を残す

## 状況

ジョブ行は最新の状態しか持たず,何回目の試行がいつ何で落ちたかが残らない

consumerは例外を`console.error`に出して捨てていた
`report`の引数も成否のbooleanだけで,失敗の理由がDOに届く経路が存在しなかった

ダッシュボードから「なぜ5回失敗したのか」が読めない

## 決定

DOに`attempt`表を置き,試行1回ぶんを記録する

| 列 | 内容 |
|---|---|
| `state` | その試行の結果 |
| `started_at` | 遷移前の`dispatched_at` |
| `finished_at` | 報告を受けた時刻 |
| `error` | 例外の文字列 |

consumerは`report(jobId, { ok: false, error })`で本文を渡す

D1へはジョブのスナップショットに同梱して運ぶ
別経路にすると冪等性の番人をもう1つ作ることになり,`seq`1つで済まなくなる

## 帰結

**1回目で成功した試行は記録しない**

その1件はジョブ行から完全に導出できる(`state` / `attempts` / `dispatched_at` / `updated_at`)
常時記録すると1ジョブあたりのDO書き込みが7回から8回に増える
導出できないのは失敗の理由と試行ごとの時刻だけなので,そこにだけ書き込みを払う

表示は`attemptsOf`がジョブ行から組み立てて返すので,画面上は成功も失敗も同じ形で並ぶ

**記録は状態遷移より先に行う**

`compareAndSet`がアウトボックスへスナップショットを積むので,後に記録すると履歴が同じ便に載らない
報告を受け付けられる状態かは`report`が自分で確かめる

**エラー本文は2,000字で打ち切る**

performerの例外はHTMLページ丸ごとのこともあり,無制限だとDOとD1の両方を圧迫する
1ジョブあたりに残す試行も20件までにする

**履歴は詳細でだけ返す**

一覧に載せると1画面で数百KBになり得る

**掃除はジョブ行より先に履歴を落とす**

順序を逆にすると対象を引けなくなり孤児が残る
