# ADR-0027:終端ジョブの保持を役割で分ける

## 状況

DOに終端ジョブを残す時間は`sweepAfterMs`の1つだけで,既定は5分だった

この値は`COMPLETED`を念頭に選んでいる
明細はD1へ投影済みで,済んだジョブをDOに残す理由が無いため短くてよい

しかし同じ数字が`FAILED`と`STALLED`にも掛かっていた
この2つはADR-0012で「終端だが不可逆ではない」と定めた**再開地点**であり,役割が違う

実環境で確認した帰結

```
sweep前のretry: 200   DOに行が在る
sweep後のretry: 409   DOから消えた
D1の明細:      200   一覧には出続ける
```

一覧に見えているのにRetryが通らない状態になる
手動リトライは人手の判断を前提にした機能で,気づくまでに数分から翌朝までかかる
5分の窓では「依存先が20分落ちていたので復旧後にまとめて再実行する」という最も典型的な用途が成立しない

## 決定

保持を2本に分ける

| 対象 | 設定 | 既定 |
|---|---|---|
| `COMPLETED` / `CANCELLED` | `sweepAfterMs` | 5分 |
| `FAILED` / `STALLED` | `failedRetentionMs` | 7日 |

失敗側の既定はD1の読み取りモデルの保持と揃える
揃えることで**一覧に見えるジョブはリトライできる**という不変条件が成立する

## 帰結

スケジューラは`state IN ('SCHEDULED','QUEUED','RUNNING')`しか読まず,索引`job_active`が終端行を跨ぐ
失敗行を長く抱えてもtickのホットパスに費用は乗らない,増えるのはストレージだけ

sweepのalarmは一定間隔ではなく**次に対象が出る時刻**を計算して張る
一定間隔のままだと失敗ジョブだけが残る間ずっと何もしない書き込みが積まれる

削除条件と次回時刻の計算は同じSQL片(`SWEEPABLE`)を共有する
片方だけ直すと消える条件と起きる条件がずれる

**残る懸念**: 失敗し続けるbindingでは7日分が積まれる
1KBのペイロードで毎秒10件失敗し続けると6GB程度になり,1 DOあたり10GBの上限に触れ得る
件数上限は入れていない,保持期間だけで足りるかは運用を見て判断する

**D1側の保持を変える場合は両方を合わせること**
片方だけ延ばすと再び「見えるのに再開できない」状態に戻る
